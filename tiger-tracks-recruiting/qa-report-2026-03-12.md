## Issues by Severity

### 🔴 CRITICAL

**C1 — DoP and DoM role parameters load but show wrong questions**  
- Section: 1a URL Routing / 8 Role Completeness  
- Roles affected: DoP, DoM  
- What's wrong: `?role=DoP` and `?role=DoM` return HTTP 200 (page loads) but neither role has a question bank in `SCREEN_QUESTIONS`. Both silently fall back to BD (Director of Partnerships) questions. A Director of Marketing candidate would be screened with BD/sales questions.  
- Fix: Add `DoP` and `DoM` entries to `SCREEN_QUESTIONS`, OR map `DoP` → `BD` and `DoM` → `MKT` explicitly so the fallback is intentional and documented. Also add both to `ROLES_DISPLAY`. Currently `ROLES_DISPLAY` has no entry for DoP or DoM.

**C2 — Videos are not reliably stored after submission**  
- Section: 4 Data Flow  
- Roles affected: ALL  
- What's wrong: The screen records video blobs in the browser and attempts to POST them to `/api/screening-submit`. However, Vercel serverless functions have a 4.5MB body size limit by default. A 5-question screening with 2–3 min responses easily exceeds 50–200MB. The server-side handler also does not process video blobs — it only reads text metadata (`candidate`, `email`, `role`, `questions`). On failure, the code falls back to localStorage — but localStorage can only hold ~5MB and cannot store video blobs, only text metadata. The actual video recordings are effectively lost after the session ends.  
- Fix (options): (a) Integrate a direct-to-S3 or Cloudinary upload for video blobs before submission — generate pre-signed URLs server-side and upload from the browser directly; (b) Use a third-party video screening service (Willo, Spark Hire) for video storage; (c) If video storage is out of scope for now, update the UI to clearly tell candidates their video is NOT being saved — otherwise this is misleading.

### 🟠 MAJOR

**M1 — No graceful fallback for missing or invalid ?role= parameter**  
- Section: 1b Edge Cases  
- What's wrong: Accessing `screen.html` with no role param, an empty param, or an invalid value silently defaults to BD (Director of Partnerships) questions. A candidate who receives a broken link or accesses the base URL would be shown the wrong role's screening. No error message, no role selector.  
- Fix: Add a validation check at page load — if `roleKey` is not in `SCREEN_QUESTIONS`, show a friendly error: "This link appears to be incomplete. Please contact recruiting@tigertracks.ai for a valid screening link."

**M2 — Role enforcement is client-side only**  
- Section: 7 Security  
- What's wrong: Any candidate can manually edit the URL from `?role=AC` to `?role=BD` and see director-level questions. Submissions are tagged with whatever `roleKey` is in the URL at the time, so a candidate could also submit under a wrong role.  
- Fix: For genuine security, implement token-based URLs (`?token=abc123`) where the token maps server-side to a specific candidate + role. For now at minimum, document this as an accepted risk since questions are not confidential material.

**M3 — Naming inconsistency: "AD" vs "DoP" / "DoM"**  
- Section: 8 Role Completeness  
- What's wrong: The QA checklist and the recruiting tool reference roles as "DoP" (Director of Partnerships) and "DoM" (Director of Marketing). The question bank uses "BD" for partnerships and "MKT" for marketing. The `ROLES_DISPLAY` maps `BD` → "Director of Partnerships, Performance" and `MKT` → "Marketing Director". There is also an `AD` key (Associate Director) with 5 questions that doesn't appear in `ROLES_DISPLAY` at all. No candidate-facing links should use `?role=AD` without a display name.  
- Fix: Decide on canonical role keys. If DoP and DoM are the intended keys, rename BD→DoP and MKT→DoM (or add aliases). Add AD to ROLES_DISPLAY or remove it.
